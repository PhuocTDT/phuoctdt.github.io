Troubleshooting

5.11.3 Troubleshooting

A categorised runbook for the errors you will actually see while building and running NutriTrack. Each entry follows the same three-line structure:

  • Symptom — the exact error string or observable behaviour
  • Root cause — what is actually wrong
  • Fix — the command to run or the file to edit

Scan by category first. If your error is not listed, jump to the debugging checklist at the bottom.

Amplify sandbox

Error: UserPoolClient already exists

  • Symptom: npx ampx sandbox fails during CFN deploy with a duplicate resource error on the Cognito User Pool Client.
  • Root cause: a previous sandbox run left partial state in CloudFormation. The local .amplify/ directory and the CFN stack have drifted.
  • Fix:
cd backend
npx ampx sandbox delete
npx ampx sandbox

If sandbox delete itself fails, delete the CFN stack manually in the AWS console (look for amplify-<app>-<user>-sandbox-*), then retry.

The provided credentials are expired

  • Symptom: sandbox or ampx generate outputs fails with an STS credentials error.
  • Root cause: your SSO session token has expired.
  • Fix:
aws sso login --profile <profile-name>

Confirm the current identity before retrying:

aws sts get-caller-identity --profile <profile-name>

sandbox secret not found for key GOOGLE_CLIENT_ID

  • Symptom: sandbox deploy fails because auth/resource.ts references a secret that was never set.
  • Root cause: secrets are per-sandbox and do not sync from another developer’s workstation.
  • Fix:
npx ampx sandbox secret set GOOGLE_CLIENT_ID
npx ampx sandbox secret set GOOGLE_CLIENT_SECRET
npx ampx sandbox secret list

Paste the value from your Google Cloud Console OAuth 2.0 client when prompted.

amplify_outputs.json not found in the frontend

  • Symptom: Metro or Expo fails to start with a missing module error on amplify_outputs.json.
  • Root cause: the file is generated by the backend and not committed.
  • Fix:
cd backend
npx ampx generate outputs --outputs-out-dir ../frontend

Re-run whenever the backend schema changes.

Bedrock

AccessDeniedException: You don't have access to the model with the specified model ID

  • Symptom: aiEngine invocations return an AccessDenied error referencing qwen.qwen3-vl-235b-a22b.
  • Root cause: model access has not been granted for your account in ap-southeast-2. Bedrock gates every foundation model behind an explicit opt-in.
  • Fix: AWS Console → Bedrock → Model access → Request access → select Qwen3-VL 235B A22B in ap-southeast-2 → submit. Propagation takes minutes to hours. Re-invoke once the status flips to Access granted.

ThrottlingException: Rate exceeded

  • Symptom: intermittent throttles under load, usually during image analysis bursts.
  • Root cause: on-demand Bedrock has per-account, per-model, per-region rate limits.
  • Fix: add exponential backoff in callQwen() (retry on ThrottlingException, ModelTimeoutException, ServiceUnavailableException). For sustained traffic, request Provisioned Throughput via AWS Support.

ValidationException: Input is too long

  • Symptom: image analysis fails when the user uploads a very large photo.
  • Root cause: base64-encoded image plus prompt exceeds the model’s context window.
  • Fix: the resizeImage Lambda already caps the longest side at 1280 px before the image reaches aiEngine. Verify that the handler reads from media/ (post-resize) and not incoming/ (pre-resize) for the analysis path. If the path is correct, reduce the max dimension to 1024 px.

Wrong-region errors on Bedrock

  • Symptom: The model ID is not supported in this region or similar.
  • Root cause: the Bedrock client or Lambda environment was configured for a region where qwen.qwen3-vl-235b-a22b is not approved.
  • Fix: hard-code REGION = "ap-southeast-2" in ai-engine/handler.ts and confirm the IAM resource ARN matches. Do not rely on AWS_REGION — the Lambda can run in any region and still need to call Bedrock in Sydney.

DynamoDB

ResourceNotFoundException: Requested resource not found in processNutrition

  • Symptom: nutrition lookups fail in sandbox when multiple developers share an AWS account.
  • Root cause: discoverTableName() uses ListTables() and picks the first Food-* name it finds — which can belong to another sandbox.
  • Fix: inject the table name explicitly in backend/amplify/backend.ts:
const cfnProcessNutritionFn = backend.processNutrition.resources.cfnResources.cfnFunction;
cfnProcessNutritionFn.addPropertyOverride(
  'Environment.Variables.FOOD_TABLE_NAME',
  backend.data.resources.tables['Food'].tableName
);

The handler already honours process.env.FOOD_TABLE_NAME first. Re-deploy, then remove dynamodb:ListTables from the IAM statement.

Same bug class on friendRequest

  • Symptom: friendRequest Lambda throws USER_TABLE_NAME env var not set or writes to the wrong Friendship table.
  • Root cause: env var injection block missing or stale after a schema rename.
  • Fix: friendRequest already has the correct pattern in backend.ts. If you renamed the user or Friendship model in data/resource.ts, update the key in backend.data.resources.tables['<name>'] to match. Both USER_TABLE_NAME and FRIENDSHIP_TABLE_NAME must resolve.

ConditionalCheckFailedException on friendRequest

  • Symptom: sendRequest or acceptRequest fails with a conditional check error.
  • Root cause: the friendship row already exists (duplicate request, or the other side already accepted).
  • Fix: catch ConditionalCheckFailedException in the Lambda and return a typed response ({ success: false, code: 'ALREADY_FRIENDS' }). The frontend should render a friendly message, not a crash.

Lambda / Node.js 22

Cannot find module 'sharp' in resizeImage

  • Symptom: resizeImage invocations crash at import time.
  • Root cause: sharp ships native binaries per OS/CPU. When installed on macOS or Windows, the Lambda bundle does not contain the Linux ARM64 binary.
  • Fix: install sharp with the correct target:
cd backend/amplify/resize-image
npm install --os=linux --cpu=arm64 sharp

Alternatively, package sharp as a Lambda layer and reference it from the function. Verify the binary exists under node_modules/@img/sharp-linux-arm64/ before deploying.

Cold start exceeds 2 seconds

  • Symptom: first invocation after idle is slow; subsequent calls are fast.
  • Root cause: Node.js 22 with several SDK clients has a non-trivial init. Memory setting also bounds CPU.
  • Fix: raise memoryMB in the function’s resource.ts (256 → 512 or 1024). For user-facing hot paths (aiEngine), consider Provisioned Concurrency instead of raising memory indefinitely.

Timeout on aiEngine during long Bedrock calls

  • Symptom: voice-to-food or image analysis returns a Lambda timeout after 30 seconds.
  • Root cause: Bedrock can take 10–60 s on long prompts or cold model routing. The default Lambda timeout is too low.
  • Fix: set timeoutSeconds: 120 in backend/amplify/ai-engine/resource.ts. If the underlying Transcribe + Bedrock chain legitimately exceeds that, redesign as two invocations (start job → poll via AppSync subscription) rather than raising timeout further.

Cognito

UserNotFoundException on sign-in

  • Symptom: login fails immediately with UserNotFoundException.
  • Root cause: the email address is not registered in this User Pool.
  • Fix: route the user to the signup screen. Do not surface the raw error — it leaks which emails exist.

UserNotConfirmedException

  • Symptom: login fails because the email OTP was never verified.
  • Root cause: signup completed but the verification step was skipped.
  • Fix: call resendSignUpCode({ username }) and navigate to the verify-otp screen. The user enters the new code and confirms.

NotAuthorizedException: Incorrect username or password

  • Symptom: login fails even though the user exists.
  • Root cause: wrong password, or the account was disabled in the Cognito console.
  • Fix: offer “Forgot password” flow. If the issue is admin-side (disabled account), check the user’s status in the Cognito console.

Google OAuth redirect loop

  • Symptom: Google login bounces between Cognito hosted UI and Google, never landing on the app.
  • Root cause: callback URL mismatch. Cognito’s allowed callback URLs and Google Cloud Console’s authorised redirect URIs must match exactly, including scheme, host, path, and trailing slash.
  • Fix: in Cognito → App client settings, copy the callback list. Paste it verbatim into Google Cloud Console → OAuth 2.0 Client → Authorised redirect URIs. Remove any legacy dev URLs that no longer exist.

Expo / React Native

Peer-dependency errors on npm install

  • Symptom: npm install in frontend/ fails with ERESOLVE peer conflicts.
  • Root cause: Expo’s native-module ecosystem has strict peer ranges that newer React Native versions violate.
  • Fix: always use --legacy-peer-deps. This is enforced in frontend/.npmrc; if you delete it, re-create it:
legacy-peer-deps=true

Metro cannot resolve amplify_outputs.json

  • Symptom: Metro bundler errors on import of amplify_outputs.json.
  • Root cause: the file is not checked in and was not generated locally.
  • Fix: regenerate from the backend:
cd backend
npx ampx generate outputs --outputs-out-dir ../frontend

Biometric prompt does not appear

  • Symptom: expo-local-authentication never prompts on Android or iOS.
  • Root cause: Expo Go does not ship with every native module. Biometric support requires a dev build.
  • Fix:
cd frontend
npx expo run:android      # or
npx expo run:ios
# For a hosted build:
eas build --profile development --platform android

Reinstall the resulting dev client and the prompt will appear.

ECS (FastAPI side)

Task stuck in PROVISIONING

  • Symptom: ECS tasks never reach RUNNING; ECS service events show repeated pull failures.
  • Root cause: no network path to ECR (missing NAT Gateway for private subnets), or the task execution role lacks ECR permissions.
  • Fix: confirm AmazonECSTaskExecutionRolePolicy is attached to the execution role (ecr:GetAuthorizationToken, ecr:BatchGetImage). If using private subnets, ensure a NAT Gateway or ECR interface endpoints exist.

Task stops immediately after start

  • Symptom: task transitions PROVISIONING → RUNNING → STOPPED in seconds.
  • Root cause: the container crashes on startup — missing env var, failed DB connection, syntax error.
  • Fix: open CloudWatch Logs for the task’s log group. The first 10 lines after startup usually name the exact issue. Add missing env vars via the task definition, not the image.

ALB returns 502 Bad Gateway

  • Symptom: ALB routes traffic but every request is 502.
  • Root cause: health check path (default /health) is not implemented in the FastAPI app, so the target group marks tasks unhealthy.
  • Fix: add a simple handler:
@app.get("/health")
def health():
    return {"status": "ok"}

Rebuild, push, redeploy. Confirm the target group’s health check path in the console.

Generic debugging checklist

When nothing on this page matches, work through the list in order — cheapest first:

  1. Reproduce locally with npx ampx sandbox pointed at your own account. Most “mysterious” bugs become boring when you own the stack.
  2. Check CloudWatch Logs for the specific Lambda. Set DEBUG=true as an env var on the function to enable the handler’s debug() logger.
  3. Set CloudWatch log retention on every log group (default is Never expire, which is both expensive and unhelpful). Two weeks is a good default for non-production.
  4. Tag every resource with Project=NutriTrack and Environment=sandbox|staging|prod. Makes cost tracking and cleanup trivial.
  5. Run the AWS CLI with --debug when a call behaves unexpectedly. The full signed request and raw response help identify policy vs. configuration issues.
  6. Compare IAM statements against 5.11.2 IAM Policies. If a statement is missing, no amount of code-level retries will save you.
  7. Check the region twice. NutriTrack pins Bedrock to ap-southeast-2 — every other service can be regional, Bedrock is not.

If you are still stuck after the checklist, capture the error, the exact command, the region, and the Lambda request ID, and file an issue.
